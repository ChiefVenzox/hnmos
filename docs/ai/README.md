# HNMos AI Integration

## Status

HNMOS-CODEX-15 started the AI-ready abstraction layer. AI Studio now adds a working OpenAI provider path for the QEMU development environment.

No LLM is embedded in the kernel. HNMos frames bounded HNLang tasks over COM2; a host bridge performs the HTTPS Responses API call and returns an untrusted draft. Native networking, GPU inference, and a model loader are still not part of Kernel 01.

HNLang generation uses the external bridge. System status/help requests remain on the deterministic local provider so raw kernel context is not sent to the external model.

## Why Real LLM Inference Is Not In The Kernel Yet

HNMos is a bare-metal operating system. A full LLM runtime would require substantial memory management, scheduling, model loading, numerical kernels, device support, and safety policy work.

Putting a full model directly inside the early kernel would be unsafe and brittle because:

- Kernel heap and memory ownership are still early.
- There is no userspace isolation yet.
- There is no filesystem-backed model loader.
- There is no networking stack.
- There is no GPU or accelerator driver layer.
- Blocking the kernel while waiting for AI output would break responsiveness.

The implementation keeps fixed task structures and provider interfaces, with a COM2 transport adapter that can later be replaced by native user-space networking.

The first implementation uses one static task slot in the bridge. It does not allocate memory dynamically and does not wait in a blocking loop.

## AI Bridge Architecture

```text
terminal/UI/HNlang request
  -> ai_bridge
  -> active ai_provider
  -> ai_task status/result
  -> console/UI output
```

The bridge owns the active provider pointer and last task slot.

Current HNLang bridge behavior:

1. Prepare a task with fixed buffers.
2. Attach the compact HNLang grammar/security profile.
3. Submit a framed request over COM2.
4. IRQ3 wakes the kernel when the response arrives.
5. Validate the `/workspace/main.hn` target through policy.
6. Store and render the untrusted RAM draft.

There is no blocking wait loop. The provider response is asynchronous and cannot directly execute shell, syscall, driver, kernel, boot, policy, or audit operations.

## AI Assembly And Machine Snapshot

The AI bridge now builds a small bounded "AI assembly" program for each AI task.

This is not executable x86 code generated by AI. It is a deterministic internal instruction plan:

```text
observe_machine
route_context
bound_task
draft_response
wait_permission
return_result
```

For `ai sync`, the plan uses explicit sync instructions:

```text
observe_machine
sync_cpu
sync_ram
sync_gpu
route_context
return_result
```

The `observe_machine` step is backed by tiny i386 assembly helpers that read only sanitized CPU state:

```text
EFLAGS interrupt flag
CR0 protected-mode bit
CR0 paging bit
```

Raw kernel memory, physical addresses, page tables, driver buffers, and generated machine code are not exposed to AI.

The goal is to make the AI integration talk to the machine through a structured OS-owned plan instead of free-form text.

The sync path is backed by the System Sync Layer. It serializes CPU state, applies a RAM memory barrier, and publishes a CPU-side framebuffer write barrier when a framebuffer is available. It does not prove vblank, scanout, or GPU command completion, expose raw GPU queues, or let AI execute hardware commands.

```text
AI Stub: provider active. Real AI runtime is not enabled yet.

AI Stub: AI assembly plan ready. Kernel snapshot observed; external actions remain gated.

ai sync
AI Stub: Kernel-owned CPU/RAM/framebuffer checkpoint executed; external actions remain gated.
```

These strings are not generated by an LLM.

## HNlang Integration Plan

HNlang Stage 0 already reserves AI task syntax:

```hn
ai task "summarize_system" {
  input system.info
  mode "local_or_bridge"
  output console
}
```

Future flow:

```text
HNlang parser
  -> ai task AST
  -> permission/safety check
  -> ai_bridge submit
  -> provider result
  -> console/UI/audit output
```

HNlang AI tasks must be reviewable by default. They must not directly call kernel memory, ports, interrupts, framebuffer writes, or filesystem writes.

## Security And Safety Boundaries

Current AI layer cannot:

- Read raw kernel memory.
- Write raw kernel memory.
- Access hardware ports.
- Modify interrupts.
- Modify scheduler state.
- Write directly to framebuffer memory.
- Access filesystems for writes.
- Use networking.
- Load models.
- Run arbitrary generated code.
- Execute AI-generated assembly or machine instructions.
- Control CPU, RAM, or GPU outside the System Sync Layer.

Allowed early behavior:

- Return fixed stub text.
- Report provider status.
- Accept bounded input text into a static task slot.
- Expose a future-safe provider abstraction.

## Roadmap

```text
Phase 1: stub provider
Phase 2: HNlang AI task parser
```
