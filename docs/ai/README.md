# HNMos AI Integration Skeleton

## Status

HNMOS-CODEX-15 adds the first AI-ready abstraction layer for HNMos.

This is not a real AI runtime. No LLM is embedded in the kernel. No networking, GPU, accelerator, model loader, or external AI dependency is added.

Current behavior is deterministic stub output only. Every placeholder response is labeled with `AI Stub:`.

## Why Real LLM Inference Is Not In The Kernel Yet

HNMos is a bare-metal operating system. A full LLM runtime would require substantial memory management, scheduling, model loading, numerical kernels, device support, and safety policy work.

Putting a full model directly inside the early kernel would be unsafe and brittle because:

- Kernel heap and memory ownership are still early.
- There is no userspace isolation yet.
- There is no filesystem-backed model loader.
- There is no networking stack.
- There is no GPU or accelerator driver layer.
- Blocking the kernel while waiting for AI output would break responsiveness.

The correct first step is a small AI bridge with fixed data structures, provider interfaces, and stub responses.

## AI Task Model

`struct hnm_ai_task` contains:

```text
task_id
task_type
input text buffer
context_flags
status
output buffer
```

Task statuses:

```text
AI_TASK_EMPTY
AI_TASK_PENDING
AI_TASK_RUNNING
AI_TASK_DONE
AI_TASK_ERROR
```

The first implementation uses one static task slot in the bridge. It does not allocate memory dynamically and does not wait in a blocking loop.

## Provider Interface

Each provider exposes:

```text
provider name
provider type
submit task
poll task
get result
```

Provider types planned:

```text
local_tiny_model
external_bridge
hardware_accelerator
offline_rule_engine
```

Current provider:

```text
stub
```

## AI Bridge Architecture

```text
terminal/UI/HNlang request
  -> ai_bridge
  -> active ai_provider
  -> ai_task status/result
  -> console/UI output
```

The bridge owns the active provider pointer and last task slot.

Current bridge behavior:

1. Prepare a task with fixed buffers.
2. Submit it to the stub provider.
3. Poll once.
4. Return the deterministic placeholder result.

There is no wait loop and no asynchronous model runtime yet.

## Stub Provider Responses

```text
ai help
AI Stub: Future HNMos assistant will help with system tasks, scripts, and diagnostics.

ai sysinfo
AI Stub: Kernel status summary is available. Real AI runtime is not enabled yet.

AI Stub: provider active. Real AI runtime is not enabled yet.
```

These strings are not generated by an LLM.

## HNlang Integration Plan

HNlang Stage 0 already reserves AI task syntax:

```hn
ai task "summarize_system" {
  input system.info
  mode "local_or_bridge"
  output console
}
```

Future flow:

```text
HNlang parser
  -> ai task AST
  -> permission/safety check
  -> ai_bridge submit
  -> provider result
  -> console/UI/audit output
```

HNlang AI tasks must be reviewable by default. They must not directly call kernel memory, ports, interrupts, framebuffer writes, or filesystem writes.

## Security And Safety Boundaries

Current AI layer cannot:

- Read raw kernel memory.
- Write raw kernel memory.
- Access hardware ports.
- Modify interrupts.
- Modify scheduler state.
- Write directly to framebuffer memory.
- Access filesystems for writes.
- Use networking.
- Load models.
- Run arbitrary generated code.

Allowed early behavior:

- Return fixed stub text.
- Report provider status.
- Accept bounded input text into a static task slot.
- Expose a future-safe provider abstraction.

## Roadmap

```text
Phase 1: stub provider
Phase 2: HNlang AI task parser
Phase 3: external dev bridge for QEMU experiments
Phase 4: local tiny inference runtime
Phase 5: accelerator/driver integration
```

Phase 3 must remain a development bridge, not a runtime dependency for booting HNMos.
